WordPress — User Manual

Customer Service image with WordPress 7.0.0, PHP 8.3 FPM, MariaDB 11.4.8, Nginx 1.30.3, and WP-CLI 2.12.0

Overview

This image starts WordPress automatically on Ubuntu 26.04. On first boot, the stack is ready and you finish the WordPress browser wizard yourself. The image keeps the database and application files on Docker volumes so the site survives reboots and normal maintenance.

Stack

  • WordPress 7.0.0 — application and PHP runtime
  • PHP 8.3 FPM — PHP processing behind Nginx
  • MariaDB 11.4.8 — database for posts, users, and settings
  • Nginx 1.30.3 — web server and reverse proxy
  • WP-CLI 2.12.0 — command-line recovery and admin tasks

What this image is for

  • Quick browser-based WordPress setup
  • Simple helper commands for daily maintenance
  • Offline-safe first boot after the image is built
  • Easy recovery after IP or domain changes

Credentials

Database credentials file

The image writes database credentials to this file on first boot:

/root/wordpress-credentials.txt

View the file

cat /root/wordpress-credentials.txt

What is inside

The file contains database connection details for the local WordPress stack. It does not contain your WordPress admin password.

Reset an admin password

Use the wp-cli helper from the shell if you need to reset the WordPress admin password.

cd /opt/wordpress
wp-cli user list --role=administrator
wp-cli user update <admin-username> --user_pass='<new-strong-password>'
Warning: Do not share the contents of /root/wordpress-credentials.txt. It contains database secrets.

HTTPS

1

Point your domain to this VM IP.

2

Place your certificate files in the cert directory:

/opt/wordpress/certs/fullchain.pem
/opt/wordpress/certs/privkey.pem
3

Set safe file permissions:

chmod 644 /opt/wordpress/certs/fullchain.pem
chmod 600 /opt/wordpress/certs/privkey.pem
4

Switch from the HTTP profile to the HTTPS profile:

cd /opt/wordpress
docker compose --profile http stop nginx
docker compose --profile https up -d nginx-https
5

Confirm the secure site loads in the browser and with a quick HTTP check.

Note: If you later change the domain, update the WordPress URLs too. See URL Recovery.

Backup & Restore

Back up the database

cd /opt/wordpress
source .env
docker compose exec db mysqldump -u root -p"$MARIADB_ROOT_PASSWORD" wordpress > db-backup.sql

Back up files and config

cd /opt/wordpress
tar czf wp-files-backup.tar.gz .env nginx/ php/ docker-compose.yml

Restore the database

cd /opt/wordpress
source .env
cat db-backup.sql | docker compose exec -T db mysql -u root -p"$MARIADB_ROOT_PASSWORD" wordpress

Restore files and config

cd /opt/wordpress
tar xzf wp-files-backup.tar.gz
Warning: Never run a full data-volume delete on a live customer site unless you intend to remove the site data.

URL Recovery

WordPress stores the site URL in the database. If the VM IP or domain changes, update both URL values.

If you can still sign in

  1. Open Settings → General in WordPress admin.
  2. Update WordPress Address (URL) and Site Address (URL).
  3. Save the changes.

If admin access is broken

cd /opt/wordpress
wp-cli option update siteurl http://<NEW-VM-IP-OR-DOMAIN>
wp-cli option update home http://<NEW-VM-IP-OR-DOMAIN>
wordpress-restart
Tip: Use the same scheme you expect customers to use, such as http:// during setup or https:// after TLS is enabled.

Upgrade & Rollback

1

Take a backup first. See Backup & Restore.

2

Run the upgrade helper.

3

Check that the stack comes back cleanly.

wordpress-status

Rollback

If the upgrade causes trouble, restore the previous stack state with the rollback helper.

wordpress-rollback
wordpress-status
Warning: Upgrade and rollback affect the local stack images and config backups. Keep your own copy of important content backups before changing anything major.

Password Reset

Use this section when a WordPress admin password is lost.

Find the admin account

cd /opt/wordpress
wp-cli user list --role=administrator

Set a new password

cd /opt/wordpress
wp-cli user update <admin-username> --user_pass='<new-strong-password>'
Note: The password reset changes the WordPress user password only. It does not change database credentials.

SMTP

This image does not include a mail server. If you need password reset emails or notifications, install an SMTP plugin and connect it to your email provider.

Recommended plugin

  • WP Mail SMTP — common choice for small-business sites
  • FluentSMTP — another valid option if your mail provider supports it

Install with WP-CLI

cd /opt/wordpress
wp-cli plugin install wp-mail-smtp --activate

Finish in WordPress admin

  1. Open the plugin settings page in WordPress admin.
  2. Enter your SMTP host, port, username, password, and sender address.
  3. Send a test email before you rely on password reset mail.
Warning: Do not hardcode SMTP credentials in this manual. Store them in the plugin settings or your own secret manager.

Snapshot & Clone

A snapshot is not a fresh install. A cloned VM keeps the same WordPress database, uploads, plugins, themes, and site URL from the source VM.

What to expect

  • The new VM has the same site content as the source VM.
  • The new VM gets a new VM IP.
  • WordPress may still point to the old URL until you update it.

If you want a fresh site

  1. Use a clean image instead of a snapshot clone.
  2. Or remove the runtime stack data before the next first boot.
  3. Then complete the wizard again from scratch.
Warning: A snapshot clone is useful for copy-and-test workflows, but it is not safe to treat it as a brand-new WordPress site without checking the database and site URL first.
